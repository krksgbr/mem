# transcript-browser

## Overview
- **Repo Name:** `transcript-browser`
- **Category/Family:** TUI / Session Manager / TUI Framework Prototype
- **Purpose:** A terminal user interface to browse, filter, and read local AI agent conversation transcripts. It serves as a strict implementation of the Model-View-Update (Elm Architecture) pattern, explicitly designed to separate core logic from terminal rendering for high testability.
- **Provider Coverage:** Currently only supports Claude Code (`Agent::Claude`).

## Ingestion, Storage, and Update Strategy
- **Ingestion:** Reads `.jsonl` files directly from the `~/.claude/projects/` directory. It uses simple string replacement heuristics to decode Claude's flattened directory names back into standard file paths.
- **Storage Strategy:** Zero persistent database. It operates entirely in-memory, holding the full hierarchical state (`Model -> projects -> conversations -> messages`) loaded at startup.
- **Update Mechanism:** There is no background watcher daemon, polling mechanism, or live reload. Files are synchronously read exactly once before the `crossterm` event loop begins. To see new messages, the application must be fully restarted.
- **Indexing:** Minimal. There is no full-text search. It relies on in-memory traversal and basic filtering based on an `Agent` enum (currently only filtering by `Claude`).

## Transcript Structure Model
The internal data model simplifies and flattens the rich source material. It relies on a three-tier hierarchy defined in `shared/src/app.rs`:

```rust
pub struct Project {
    pub name: String,
    pub conversations: Vec<Conversation>,
}

pub struct Conversation {
    pub id: String,
    pub agent: Agent,
    pub messages: Vec<Message>,
}

pub struct Message {
    pub role: String,
    pub content: String,
}
```

- During parsing, Claude's complex JSON structures (e.g., arrays of blocks, tool uses, thinking blocks) are flattened into a single `String` representing the `content` of a `Message`. It drops metadata like timestamps and tool execution IDs.
- The `Conversation` ID is dynamically generated by attempting to find a `slug` in the JSONL stream, falling back to a truncated preview of the first user message.

## UX Shape
- **Built with:** Rust, `ratatui` (TUI), `crossterm`, and the `crux` framework.
- **Navigation:** Operates on a three-pane drill-down model based on the `Screen` enum state:
  1. **Projects List**
  2. **Conversations List** (filtered by agent, e.g., "F to cycle")
  3. **Messages View** (ledger-style reading)
- **UI Elements:** Features a persistent header with breadcrumbs (e.g., `Projects > ProjectName > ConvID`) and a footer displaying keyboard shortcuts.
- **Keybindings:** Standard vim-like keys (`j/k` for up/down, `h/l` or `Enter/Esc` for drill-in/out).
- **Styling:** Uses a custom dark theme palette distinguishing User messages (dark gray background, white text) from Assistant messages (darker gray background, dimmed text) using a visual padding strategy.

## Strengths
- **Impeccable Testability:** By enforcing the strict `crux` Model-View-Update architecture, all state transitions and filtering logic are decoupled from the terminal and can be exhaustively tested via standard unit tests (`AppTester`).
- **Clean Rendering Tests:** The UI layer is designed to be validated through Ratatui's `TestBackend`, allowing assertions on visual buffer output without ever running a blocking terminal session.
- **Fast Traversal:** Because everything is loaded synchronously into a pure Rust memory model on startup, navigating between lists and messages is instant.

## Weaknesses
- **Synchronous Startup:** Parsing the entire `~/.claude/projects/` directory synchronously blocks the UI thread. This will scale extremely poorly as the user's transcript history grows.
- **Lossy Data Model:** Flattening all Claude log entries into simple `role`/`content` strings discards critical context like "Thinking" visibility, nested tool arguments, and tool results, severely limiting the depth of the reading experience.
- **No Live Updates:** The lack of file watching makes it purely a post-mortem viewer rather than an active companion tool.

### Flattened Ingestion Parser
*Demonstrates how complex JSON messages are recursively flattened into a single text block, discarding structured blocks.*
```rust
// cli/src/main.rs
if let Some(msg) = val.get("message") {
    let role = msg.get("role").and_then(|v| v.as_str()).unwrap_or("unknown").to_string();
    let content_val = msg.get("content");
    
    let mut text_content = String::new();
    if let Some(content) = content_val {
        if let Some(s) = content.as_str() {
            text_content = s.to_string();
        } else if let Some(arr) = content.as_array() {
            for item in arr {
                if let Some(obj) = item.as_object() {
                    if let Some(t) = obj.get("text").and_then(|v| v.as_str()) {
                        if !text_content.is_empty() {
                            text_content.push_str("\n\n");
                        }
                        text_content.push_str(t);
                    }
                }
            }
        }
    }
    // ...
}
```
