## Purpose
Fast local history search for Claude, Codex CLI, and OpenCode logs. It provides a way to index, search, and browse past agent conversations using both exact term matching (BM-25) and fuzzy conceptual matching (vector embeddings). It's primarily designed for agents to use via skills to retrieve their own history or for users to browse history via a TUI.

## Provider Coverage
- **Claude Code:** Parses session logs from standard Claude locations.
- **Codex CLI:** Supports both session-specific logs and global history logs.
- **OpenCode:** Ingests OpenCode agent logs.

## Ingestion, Storage & Update Strategy
- **Ingestion:** Uses `mmap` for high-performance reading and `rayon` for parallel processing. It tracks file offsets and modification times (`mtime`) in `ingest.json` to perform incremental updates. A `scan_cache.json` with a TTL (default 1 hour) prevents unnecessary filesystem walks.
- **Storage:** 
  - **Text Index:** Uses `tantivy` for full-text indexing (BM-25).
  - **Vector Index:** Uses `usearch` for storing and searching vector embeddings.
  - **Local Embeddings:** Embeddings are generated locally using `fastembed` with models like `minilm` or `bge`, leveraging macOS ANE/GPU when available.
- **Update Mechanisms:** 
  - Manual: `memex index`
  - On-demand: Auto-indexing during search (default).
  - Background: Background service via `launchd` (macOS) or `systemd` (Linux).

## Transcript Structure Model
Memex normalizes all sources into a flat `Record` structure:

| Field | Type | Description |
|-------|------|-------------|
| `doc_id` | `u64` | Unique document identifier. |
| `ts` | `u64` | Unix timestamp of the message. |
| `project` | `String` | Project name or path where the session occurred. |
| `session_id` | `String` | Unique session identifier. |
| `turn_id` | `u32` | Order of messages within a session. |
| `role` | `String` | `user`, `assistant`, or `tool`. |
| `text` | `String` | Main message content. |
| `tool_name` | `Option<String>` | Name of the tool if `role` is `tool`. |
| `tool_input` | `Option<String>` | Input passed to the tool. |
| `tool_output` | `Option<String>` | Output returned by the tool. |
| `source_path`| `String` | Path to the original log file. |

## UX Shape
- **CLI Interface:** Provides `search` (JSONL/Human output), `session` (full transcript dump), and `show` (single record) commands.
- **TUI Interface:** A `ratatui`-based browser with:
  - Global search/filter input.
  - Scrollable session list with summaries.
  - Detail pane showing the full conversation with tool calls.
  - Keybindings for resuming sessions in the original agent.

## Strengths
- **Local-First:** All indexing and embedding happen on-device (no API keys required for search).
- **Hybrid Search:** Combines keyword precision with semantic relevance.
- **High Performance:** Written in Rust, utilizes `mmap`, `simd-json`, and parallel ingestion.
- **Low Overhead:** Incremental indexing ensures updates are fast.
- **Deep Tool Awareness:** Explicitly models tool inputs and outputs.

## Weaknesses
- **Fixed Providers:** Hardcoded logic for specific agent log formats.
- **Resource Intensive:** Local embeddings require significant CPU/GPU/ANE for large histories.
- **Flat Schema:** Loses some hierarchical metadata (e.g., nested tool calls) by flattening into records.
