## Overview
- **Repository:** `zippoxer/recall`
- **Category:** TUI / CLI Session Manager & Full-Text Indexer
- **Purpose:** Search, review, and quickly resume AI coding assistant conversations directly from the terminal.
- **Provider Coverage:** Claude Code, Codex CLI, Factory (Droid), OpenCode.

## Architecture & Storage Strategy

### Ingestion & Indexing
- **Discovery:** Recall scans hardcoded provider-specific local paths (e.g., `~/.claude/projects/*/*.jsonl`, `~/.codex/sessions/**/*.jsonl`) using `walkdir` and `std::fs`.
- **Indexing Engine:** Uses **Tantivy** (a fast, Lucene-inspired Rust search engine library). The index is stored locally in `~/.cache/recall/index`.
- **Update Mechanism:** Recall utilizes an incremental indexing strategy tracked by a `state.json` file containing file modification times (`mtime`).
  - **TUI Mode:** Launches a background thread upon startup that discovers new or modified `.jsonl` files and indexes them asynchronously while the user interacts with the UI.
  - **CLI Mode:** Performs synchronous incremental indexing (`index/sync.rs`) before executing search queries, ensuring the JSON output for agents is always perfectly up-to-date.

### Transcript Structure Model
Recall normalizes transcripts into a highly simplified, human-readable structural model:
- Models a `Session` with metadata (source, path, cwd, branch, timestamp) and a vector of `Message` objects.
- `Message` consists of simply a `Role` (`User` or `Assistant`) and pure `String` content.
- **Lossy Parsing:** The ingestion process deliberately discards context to focus on high-signal conversational text. For example, `ClaudeParser` explicitly strips out `tool_use`, `thinking` blocks, internal slash-command expansions (`<command-message>`), and synthetic messages (e.g., compaction summaries like `isCompactSummary`).
- **Compaction:** Consecutive messages belonging to the same role are grouped and concatenated together using `join_consecutive_messages`.

## UX Shape

Recall has a dual UX approach, catering perfectly to both human engineers and AI agents without requiring an MCP (Model Context Protocol) server.

- **Human TUI (`ratatui`):** 
  - Executing `recall` with no arguments launches an interactive search TUI.
  - "Start typing to search. Enter to jump back in."
  - Users can toggle between "folder-scoped" and "everywhere" searches by pressing `/`. Arrow keys navigate sessions, and `Enter` seamlessly boots up the respective AI provider CLI via a mapped command (e.g., `claude --resume {id}`).
- **Agent CLI:** 
  - Recall exposes headless commands (`recall search <query>`, `recall list`, `recall read <session_id>`).
  - These commands output well-structured JSON data, allowing agents (like Claude Code itself) to query the user's past conversations simply by running `recall search --help`.

## Strengths
- **Blazing Fast:** Written in Rust and backed by Tantivy, searches over massive transcript histories are practically instantaneous.
- **Privacy-First:** Fully local and offline indexing. Transcripts never leave the user's machine.
- **Agent-Ready Design:** The dual TUI/CLI interface means an agent can use `recall` as its own memory backend via shell commands, circumventing the need to build, run, and maintain a dedicated MCP server.
- **Unified Interface:** Normalizes conversations across multiple coding assistants (Claude, Codex, Factory) into a single, cohesive search index and UI.

## Weaknesses
- **Lossy Ingestion:** Discarding `tool_use` and `thinking` blocks makes Recall unsuitable for auditing exact agent behavior, tracking token usage, or replaying exact tool execution sequences. It optimizes strictly for conversational recall.
- **Brittle Discovery:** Relies heavily on the undocumented, internal storage structures (like `~/.claude/projects/`) of third-party tools, making it susceptible to breaking changes if those tools update their file formats.

```rust
fn build_schema() -> Schema {
    let mut builder = Schema::builder();

    // Stored metadata fields
    builder.add_text_field("session_id", STRING | STORED);
    builder.add_text_field("source", STRING | STORED);
    builder.add_text_field("file_path", STRING | STORED);
    builder.add_text_field("cwd", STRING | STORED);
    builder.add_text_field("git_branch", STRING | STORED);

    // Timestamp for recency sorting
    builder.add_i64_field("timestamp", INDEXED | STORED | FAST);

    // Message index within the session
    builder.add_u64_field("message_index", STORED);

    // Searchable content field
    builder.add_text_field("content", TEXT | STORED);

    builder.build()
}
```

### 2. Lossy Message Extraction (`src/parser/claude.rs`)
Notice how it specifically filters for `"text"` type blocks while ignoring tools and thinking.

```rust
fn extract_content(content: &serde_json::Value) -> String {
    match content {
        // Direct string (user messages)
        serde_json::Value::String(s) => s.clone(),

        // Array of content blocks (assistant messages)
        serde_json::Value::Array(arr) => {
            let mut texts = Vec::new();
            for item in arr {
                if let Some(obj) = item.as_object() {
                    // Only extract "text" type blocks, skip tool_use, thinking, etc.
                    if obj.get("type").and_then(|v| v.as_str()) == Some("text") {
                        if let Some(text) = obj.get("text").and_then(|v| v.as_str()) {
                            texts.push(text.to_string());
                        }
                    }
                }
            }
            texts.join("\n")
        }

        _ => String::new(),
    }
}
```

### 3. Joining Consecutive Messages (`src/parser/mod.rs`)
Simplifies the conversation structure for better indexing and rendering.

```rust
pub fn join_consecutive_messages(messages: Vec<Message>) -> Vec<Message> {
    messages.into_iter().fold(Vec::new(), |mut acc, msg| {
        if let Some(last) = acc.last_mut() {
            if last.role == msg.role {
                last.content.push_str("\n\n");
                last.content.push_str(&msg.content);
                last.timestamp = msg.timestamp; // use latest
                return acc;
            }
        }
        acc.push(msg);
        acc
    })
}
```
