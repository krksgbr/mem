# ccrider - Transcript Indexer Research

## Overview

- **Repo Name:** `ccrider`
- **Category / Family:** Session Manager, TUI Browser, MCP Server, CLI Indexer
- **Purpose:** Indexes Claude Code and Codex CLI sessions into a single searchable SQLite database. It provides an interactive Terminal UI (TUI) to browse, full-text search, and resume past coding agent sessions seamlessly. It also acts as an MCP server, granting agents the ability to search your past context.
- **Provider Coverage:** Claude Code (`~/.claude/projects/`) and Codex CLI (`~/.codex/sessions/`).

## Strategy: Ingestion, Storage, and Updates

- **Ingestion Strategy:** Discovers JSONL session files inside known provider directories (e.g., `~/.claude/projects`). Utilizes a highly concurrent Goroutine worker pool to parse session files efficiently without UI dependencies.
- **Storage Strategy:** Stores session metadata and message content in a local SQLite database utilizing the FTS5 extension.
  - Generates two virtual FTS tables: `messages_fts` (stemming for natural language) and `messages_fts_code` (no stemming for exact symbol/camelCase matching).
- **Update Mechanism:** Incremental Sync. `ccrider` checks the `mtime` and `file_size` of tracked JSONL files. For modified files, it computes a new file hash, seeks to the last known byte offset, and parses/appends only the newly appended lines. This ensures extremely fast updates even with massive session histories.

## Transcript Structure Model

The model maps closely to Claude Code's schema. It processes 5 distinct message entry types natively (`summary`, `user`, `assistant`, `system`, `file-history-snapshot`).

The parsed structures look like this:

```go
type ParsedSession struct {
        SessionID string
        Summary   string
        LeafUUID  string
        Messages  []ParsedMessage
        FilePath  string
        FileSize  int64
        FileMtime time.Time
}

type ParsedMessage struct {
        UUID        string
        ParentUUID  string
        Type        string
        Sender      string
        Content     json.RawMessage
        TextContent string
        Timestamp   time.Time
        Sequence    int
        IsSidechain bool
        CWD         string
        GitBranch   string
        Version     string
}
```

The system threads messages securely using the `ParentUUID` relation, which matches Claude Code's threading architecture.

## UX Shape

- **TUI Browser:** A comprehensive Bubbletea TUI split into standard views:
  - **Browse Mode:** Shows a list of sessions, filterable by project path or date.
  - **Search Mode:** Interactive query input mapping into the SQLite FTS engine.
  - **Session View:** Renders the conversation stream linearly in the terminal.
- **Immediate Shell Operations:** One-keypress commands (`r`) spawn native resume operations (e.g., `claude --resume <session-id>`) keeping the developer in their host shell context.
- **Headless MCP Context:** Run continuously as an MCP server (`ccrider serve-mcp`), exposing past knowledge strictly to the active coding LLM logic.

## Strengths

- **Single Binary/Fast Execution:** Written in Go with SQLite. No heavy NPM/Python footprint.
- **Incredible Search Performance:** SQLite FTS5 yields sub-100ms full-text searches across 10,000+ messages.
- **Zero-Data Loss Parsing:** 100% schema coverage preserves tool-use metadata and snapshot links rather than just grepping raw text.
- **Strict Core/Interface Architecture:** Pure business/parsing logic with thin UI wrappers ensuring testability and modular usage of core packages.
- **Incremental Efficiency:** Only reads the differential bytes when synchronizing active/ongoing sessions.
- **Smart Code Queries:** Dual tokenizers ensure natural language terms stem while exact code variables/symbols do not.

## Weaknesses

- **High Coupling to Providers:** Relies entirely on the undocumented and evolving internal JSONL formats of Claude Code and Codex CLI. Any upstream schema change will break the parser.
- **Terminal Constraints:** Purely terminal-bound rendering. Visualizing large, complex diffs, images, or dense Markdown within the Bubbletea session view is inherently constrained compared to a web or Electron-based UI.

## Key Code Snippets

### Source Aggregation Strategy

```go
// internal/core/importer/sync.go
func DefaultSources() []Source {
    home, _ := os.UserHomeDir()
    sources := []Source{
        {
            Path:          filepath.Join(home, ".claude", "projects"),
            ParseFn:       ccsessions.ParseFile,
            Provider:      "claude",
            SkipSubagents: true,
        },
    }

    codexPath := filepath.Join(home, ".codex", "sessions")
    if _, err := os.Stat(codexPath); err == nil {
        sources = append(sources, Source{
            Path:          codexPath,
            ParseFn:       codexsessions.ParseFile,
            Provider:      "codex",
            SkipSubagents: false,
        })
    }
    return sources
}
```

### FTS Schema Setup

```sql
-- Natural language search with stemming
CREATE VIRTUAL TABLE messages_fts USING fts5(
    text_content,
    content=messages,
    content_rowid=id,
    tokenize='porter unicode61'
);

-- Code search without stemming (preserves symbols, camelCase)
CREATE VIRTUAL TABLE messages_fts_code USING fts5(
    text_content,
    content=messages,
    content_rowid=id,
    tokenize='unicode61'
);
```
