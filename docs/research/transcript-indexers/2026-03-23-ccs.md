# Analysis: `ccs` (Claude Code Search)

## Overview

- **Repository**: `agentic-utils/ccs` (Local path: `/Users/gaborkerekes/playground/ai-stuff/harness-utils/session-managers/ccs`)
- **Category/Family**: Transcript Indexer / Session Manager / TUI Search Tool
- **Purpose**: A globally accessible TUI to search, preview, and directly resume Anthropic's [Claude Code](https://claude.ai/claude-code) conversations.
- **Provider Coverage**: Exclusive to **Claude Code** (reads Anthropic's `~/.claude/projects/` directory).

## Strategy & Mechanisms

### Ingestion, Storage, and Update Strategy
- **On-the-fly Ingestion**: `ccs` does not maintain a persistent database or external index. Instead, it reads raw transcript files (`.jsonl`) directly from `~/.claude/projects/` into memory on startup.
- **Performance Optimization**: Because on-the-fly loading can be slow for massive logs, `ccs` employs:
  - **Worker Pools**: A pool of 8 concurrent Go routines parse the JSONL files in parallel.
  - **Heuristic Filtering**: It skips subagent logs (`agent-*.jsonl`), ignores files older than 60 days by default (`--max-age`), and ignores files larger than 1GB (`--max-size`).
- **No Incremental Updates**: The tool re-indexes everything every time it runs. This trades startup latency for architectural simplicity.

### Transcript Structure Model (Normalization)
- Claude Code emits complex JSON structures with content blocks (e.g., text, tool use, image). `ccs` extracts and normalizes this into a simpler in-memory structure focused strictly on searchable text.
- **Data Model**:
  ```go
  type Conversation struct {
      SessionID      string
      Cwd            string
      FirstTimestamp string
      LastTimestamp  string
      Messages       []Message // Role ("user"|"assistant"), Text, Ts
      FilePath       string
  }
  ```
- **Normalization**: Extracts plain text from array-based message formats and consolidates them into a single `Text` string per message, discarding structural tool blocks that aren't human-readable.

### UX Shape and Patterns
- **TUI (Text User Interface)**: Built on the popular Charm `bubbletea` and `lipgloss` stack.
- **Split View Layout**: 
  - **Top List**: A real-time filtered list of conversations containing columns for `DATE`, `PROJECT`, `TOPIC` (derived from the first user message), total `MSGS`, and search `HITS`.
  - **Bottom Preview**: A context-aware message preview that highlights search matches (yellow background) and shows surrounding message context (X messages before/after) while collapsing irrelevant parts of the transcript.
- **Interactions**:
  - Live filtering as the user types.
  - Context-aware mouse wheel scrolling (scrolls the list or the preview depending on cursor position).
  - Direct deletion of sessions (`Ctrl+D`) which actually removes the `.jsonl` file.
  - Resuming sessions by cleanly overtaking the process with `syscall.Exec` into the original working directory (`CWD`).

## Strengths & Weaknesses

**Strengths:**
- **Zero Configuration**: Relies entirely on Claude's default file structure; no daemons or sync required.
- **Excellent TUI / Preview UX**: The contextual diff/preview approach (showing matches surrounded by a few preceding/succeeding messages) is highly readable for massive chat logs.
- **Fast Enough (usually)**: Go worker pools make parsing thousands of small-to-medium files feel nearly instantaneous.
- **Seamless Resume**: Changes the working directory and replaces the current process, resulting in a perfect integration loop with the parent CLI.

**Weaknesses:**
- **Scaling Limits**: Will eventually struggle with startup latency on massive transcript repositories because it reparses all JSONL files from disk on every invocation.
- **Provider Lock-in**: Hardcoded to `~/.claude/projects/` and the Claude Code data model.
- **Naive Search**: Relies entirely on exact substring matches (case-insensitive); no lexical tokenization, stemming, or semantic search.

### 1. Parallel Ingestion (Worker Pool)
```go
// Worker pool to limit concurrent file operations
const numWorkers = 8
jobs := make(chan string, len(files))
results := make(chan *Conversation, len(files))

var wg sync.WaitGroup
for i := 0; i < numWorkers; i++ {
        wg.Add(1)
        go func() {
                defer wg.Done()
                for path := range jobs {
                        conv, err := parseConversationFile(path, cutoff, maxSize)
                        if err == nil && conv != nil {
                                results <- conv
                        }
                }
        }()
}
```

### 2. Normalization / Content Extraction
```go
func extractText(content json.RawMessage) string {
        if len(content) == 0 { return "" }
        // Attempt simple string
        var str string
        if err := json.Unmarshal(content, &str); err == nil { return str }

        // Attempt array of blocks
        var arr []TextContent
        if err := json.Unmarshal(content, &arr); err == nil {
                var parts []string
                for _, item := range arr {
                        if item.Type == "text" && item.Text != "" {
                                parts = append(parts, item.Text)
                        }
                }
                return strings.Join(parts, " ")
        }
        return ""
}
```
